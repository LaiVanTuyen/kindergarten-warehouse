> **Domain**: Education/Warehouse Management

## 1. Backend Architecture (Java Spring Boot)
**Source**: `backend-dev-guidelines` (Adapted for Java)

### Core Doctrine
1.  **Strict Layered Architecture**:
    *   `Controller` → `Service` → `Repository` → `Database`
    *   **Rule**: Controllers must **never** contain business logic. They only parse requests, call services, and return responses.
    *   **Rule**: Services must be framework-agnostic where possible and contain all business rules.
2.  **Validation is Mandatory**:
    *   Use **Jakarta Bean Validation** (`@Valid`, `@NotNull`, etc.) for all inputs (DTOs). Never trust client input.
3.  **Unified Error Handling**:
    *   Use `@RestControllerAdvice` (Global Exception Handler) to catch and log errors.
    *   Never swallow exceptions. If you catch it, log it or rethrow it wrapped in a custom exception.
4.  **Configuration**:
    *   Use `@ConfigurationProperties` or strict `application.yml` mapping. Avoid using `System.getenv()` directly in code.

## 2. Frontend Engineering (Angular)
**Source**: `frontend-dev-guidelines` (Adapted for Angular)

### Core Doctrine
1.  **Feature-Based Organization**:
    *   Group files by **feature**, not type.
    *   Structure: `src/app/features/student-management/` instead of `src/app/components/`.
2.  **Performance First**:
    *   **Lazy Load** heavy modules and routes. Don't bundle everything in `main.js`.
    *   Use **OnPush** change detection strategy by default for performance.
3.  **State Management & Data Fetching**:
    *   Use **Async Pipe** (`| async`) or **Signals** in templates to handle observables. Avoid manual `.subscribe()` in components to prevent memory leaks.
    *   Cache expensive HTTP calls where meaningful (e.g., using generic caching interceptors or services).
4.  **Strict TypeScript**:
    *   No `any`. Define interfaces/models for all API responses.

## 3. Database Design (MySQL)
**Source**: `database-design`

### Schema Rules
1.  **Primary Keys**:
    *   Use **UUID/ULID** for entities that might need migration or high scale (e.g., `Students`, `InventoryItems`).
    *   Use `Auto-increment` only for internal lookup tables (e.g., `Category`, `Role`).
2.  **Audit Columns** (Mandatory for every table):
    *   `created_at` (TIMESTAMP)
    *   `updated_at` (TIMESTAMP)
    *   `deleted_at` (Nullable TIMESTAMP for Soft Deletes - crucial for "Warehouse" history).
3.  **Normalization**:
    *   Normalize to 3NF by default to ensure data integrity.
    *   Denormalize **only** for proven performance bottlenecks (e.g., heavy reporting queries).

## 4. API Design (REST)
**Source**: `api-patterns`

### RESTful Standards
1.  **Resource Naming**:
    *   Use **Plural Nouns**: `/api/v1/students`, `/api/v1/warehouse-items`.
    *   Avoid Verbs in URLs (Bad: `/api/getStudents`, Good: `GET /api/students`).
2.  **HTTP Methods**:
    *   `GET`: Read
    *   `POST`: Create
    *   `PUT`: Update (Replace whole resource)
    *   `PATCH`: Update (Partial fields)
    *   `DELETE`: Remove
3.  **Standard Response Envelope**:
    *   Wrap responses in a consistent object:
        ```json
        {
          "data": { ... },
          "meta": { "page": 1, "total": 100 },
          "status": "success"
        }
        ```

## 5. Software Architecture & Code Quality
**Source**: `software-architecture`

### General Principles
1.  **Library-First**:
    *   Don't reinvent the wheel. Use existing libraries for common tasks (e.g., Apache Commons, Guava, MapStruct) before writing custom utilities.
2.  **Early Returns**:
    *   Avoid deep nesting (`if` inside `if` inside `if`). Check failure conditions first and return early.
3.  **Separation of Concerns**:
    *   Do not mix database queries in your UI layer.
    *   Do not mix UI logic in your Service layer.
4.  **Clean Code & Readability**:
    *   **NO Fully Qualified Names**: Always use `import` for classes. Avoid `com.example.DTO`, use `DTO` directly.
    *   **Lombok**: Use `@RequiredArgsConstructor` for Dependency Injection. Avoid boilerplate constructors.
